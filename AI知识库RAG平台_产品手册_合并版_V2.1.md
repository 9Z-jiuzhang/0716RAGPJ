## 1. 项目概述

### 1.1 项目名称

**AI 知识库 RAG 平台（AI Knowledge Base RAG Platform）**

### 1.2 项目背景

团队需要构建一个基于大语言模型（LLM）的智能知识库平台，实现文档的智能管理、高效检索和上下文感知的多轮对话问答。系统面向多角色用户，支持精细化的权限控制，同时提供完善的测试与监控能力，确保知识库的检索质量和系统运行稳定性。

### 1.3 产品目标

- 让访客、已注册用户和管理员以受控权限检索知识库并进行多轮问答。
- 让授权用户上传并管理常见格式的文档，完成预处理、分段、向量化和可追溯检索。
- 让管理员独立管理用户、角色、知识库、文档、模型配置及检索命中率测试。
- 让知识库变更具备版本快照、审计日志、回退能力和运行监控能力。

### 1.4 适用范围

适用于企业内部知识管理、技术文档检索、客户支持知识库、学术文献问答等场景。

### 1.5 不在第一期强制范围内

- 企业单点登录、组织架构同步、计费与多租户商业化。
- 扫描件 OCR 的高精度人工校对流程；可预留 OCR 扩展接口。
- 移动端原生应用、复杂流程审批、跨地域灾备。

## 3. 角色与权限模型

平台使用基于角色的访问控制（RBAC）。权限判定同时满足两层条件：角色拥有功能权限，且该角色被授予目标知识库的访问范围。任何问答、检索、文档预览、导出和测试都必须在服务端执行该校验。

特别说明 `qa:ask` 权限：虽然标记为全局功能权限，但实际检索范围由第二层条件（知识库可见性和授权范围）决定。未登录访客仅能检索 `visibility=public` 的知识库；已登录用户的检索范围受其角色和知识库授权共同限制。这意味着即使拥有 `qa:ask` 权限，不同用户对同一问题看到的结果可能不同。

### 3.1 角色矩阵

| 身份/角色 | 登录状态 | 问答 | 上传文档 | 管理知识库 | 用户/角色管理 | 命中率测试 | 快照回退 |
|-----------|---------|------|---------|-----------|-------------|-----------|---------|
| 访客 (guest) | 未登录 | 仅可问答公开知识库；无历史长期保存 | 否 | 否 | 否 | 否 | 否 |
| 注册用户 (user) | 已登录 | 可访问已授权知识库；支持本人多轮历史 | 有上传权限时可上传至指定知识库 | 仅本人可见范围内的基础查看 | 否 | 否 | 否 |
| 知识库维护员 (kb_admin) | 已登录 | 是 | 是 | 管理被授权知识库及其文档、分段、重建 | 否 | 可对被授权知识库测试 | 可对被授权资源回退 |
| 系统管理员 (admin) | 已登录 | 是 | 是 | 全量管理 | 是 | 是 | 是 |

### 3.2 权限模型

采用 RBAC 模型，支持灵活的角色自定义：

```
用户 (User) --N:M--> 角色 (Role) --N:M--> 权限 (Permission)
```

权限以"资源 + 动作"保存，统一使用冒号分隔标识符。角色是权限集合；用户可拥有一个或多个角色；知识库授权记录定义角色或用户可以访问的知识库范围。

### 3.3 权限清单

| 权限标识 | 说明 | 作用域 |
|----------|------|--------|
| `user:read` | 查看用户列表 | 全局 |
| `user:write` | 创建/修改/禁用用户 | 全局 |
| `role:read` | 查看角色列表 | 全局 |
| `role:write` | 创建/修改/删除角色 | 全局 |
| `kb:read` | 查看知识库 | 可按知识库隔离 |
| `kb:write` | 创建/修改/删除知识库 | 全局 |
| `kb:upload` | 上传文档到知识库 | 可按知识库隔离 |
| `kb:vectorize` | 触发重新向量化 | 可按知识库隔离 |
| `doc:read` | 查看文档列表 | 可按知识库隔离 |
| `doc:write` | 上传/修改/删除文档 | 可按知识库隔离 |
| `doc:segment` | 修改分段规则并重新分段 | 可按知识库隔离 |
| `qa:ask` | 进行知识库问答 | 全局 |
| `test:read` | 查看命中率测试结果 | 全局 |
| `test:write` | 执行命中率测试 | 全局 |
| `snapshot:read` | 查看快照历史 | 可按知识库隔离 |
| `snapshot:write` | 创建/删除快照 | 可按知识库隔离 |
| `snapshot:restore` | 执行回退 | 可按知识库隔离 |
| `model:read` | 查看模型配置 | 全局 |
| `model:write` | 修改模型配置 | 全局 |
| `system:read` | 查看系统统计和监控面板 | 全局 |
| `audit:read` | 查看操作审计日志 | 全局 |

### 3.4 知识库级别隔离

管理员可以为每个用户或角色单独配置对特定知识库的访问权限。例如：
- 用户 A 拥有"技术文档库"的 `kb:upload` 权限，但没有"财务文档库"的任何权限
- "研发部角色"可以访问知识库 A 和知识库 B，但无法访问知识库 C

权限检查时，系统按以下优先级合并用户的权限：
1. 用户直接绑定的知识库权限
2. 用户所属角色的知识库权限
3. 两项取并集（宽松策略，任一来源授权即放行）

### 3.5 权限约束

- 内置角色不可删除；变更角色或权限后记录审计日志。
- 权限校验仅在后端生效，前端隐藏菜单不能替代权限控制。
- 禁用用户后，应立即拒绝其新请求；其历史数据保留，不自动删除。

## 5. 功能模块详解

### 5.1 前端模块

#### 5.1.1 访问页面（访客端 / Guest Portal）

面向所有用户（含未登录访客）的公开页面。导航保持简洁，仅提供：

- 智能问答：对授权范围内知识库进行流式、多轮问答
- 文档上传：仅登录且具备上传权限时显示；选择目标知识库后上传并查看处理进度
- 登录、注册、个人中心：注册、登录、退出、修改个人资料、查看本人的会话与上传记录

访客进入时默认进入"智能问答"，系统只检索公开知识库；上传入口不可见。回答应展示引用来源、文档名、分段序号和置信提示，无法命中时明确说明未在授权知识库中找到依据，不得编造来源。

| 功能点 | 说明 | 权限要求 |
|--------|------|----------|
| 知识库问答 | 输入问题进行 AI 问答，支持多轮对话 | 访客及以上 |
| 文档上传 | 上传文档至指定知识库 | 需登录，且拥有目标知识库的 `kb:upload` 权限 |
| 用户注册 | 填写用户名、密码、邮箱完成注册 | 无 |
| 用户登录 | 账号密码登录 | 无 |
| 对话历史 | 查看个人的历史对话记录 | 需登录 |

**页面路由规划：**

| 路由 | 页面名称 | 说明 |
|------|---------|------|
| `/` | 首页/问答页 | 知识库问答主界面 |
| `/login` | 登录页 | 用户登录 |
| `/register` | 注册页 | 新用户注册 |
| `/history` | 对话历史 | 个人的历史对话列表 |
| `/profile` | 个人中心 | 修改个人资料、查看上传记录 |

**问答界面交互流程：**

1. 用户在输入框输入问题并发送
2. 系统识别当前会话上下文（基于 session_id）
3. 后端进行语义检索 + LLM 生成回复
4. 前端流式展示回复内容（SSE，打字机效果）
5. 支持多轮追问，上下文自动关联
6. 回答展示引用来源、文档名、分段序号和置信提示

#### 5.1.2 管理员页面（管理端 / Admin Portal）

**页面路由规划：**

| 路由 | 页面名称 | 说明 |
|------|---------|------|

### 5.2 认证、注册与用户中心

#### 5.2.1 功能说明

- **用户注册**：用户名 + 密码 + 邮箱，注册后默认分配"注册用户"角色，密码加密存储
- **用户登录**：JWT Token 认证机制，支持 Token 刷新
- **访客访问**：未登录用户以 `guest` 身份访问问答页面，只能进行问答，不能上传文档
- **用户管理（管理员）**：查看用户列表、启用/禁用用户、修改用户角色、重置密码
- 用户管理列表至少包含账号、昵称、状态、角色、创建时间、最近登录时间和操作记录入口

#### 5.2.2 用户状态

| 状态 | 说明 |
|------|------|
| 活跃 (active) | 正常使用状态 |
| 禁用 (disabled) | 管理员禁用，无法登录 |
| 待验证 (pending) | 注册后邮箱未验证（可选扩展） |

#### 5.2.3 认证流程

```
用户登录 -> 验证凭据 -> 生成 JWT (access_token + refresh_token)
  -> 前端存储 Token -> 每次请求携带 Authorization Header
  -> 中间件验证 Token -> 注入当前用户信息到 Request Context
```

- Access Token 有效期：30 分钟
- Refresh Token 有效期：7 天
- 刷新机制：access_token 过期后，使用 refresh_token 获取新的 access_token
- 登录失效处理：Token 过期或被禁用时返回 401，前端跳转登录页

- 支持新增、编辑、停用角色，为角色勾选功能权限。
- 支持为角色配置可访问知识库，以及只读、上传、维护、回退等资源级权限。
- 内置角色不可删除；变更角色或权限后记录审计日志。
- 权限校验仅在后端生效，前端隐藏菜单不能替代权限控制。

### 5.4 知识库管理

知识库是文档、分段、索引和权限的隔离边界。

#### 5.4.1 功能点

| 功能 | 说明 |
|------|------|
| 创建知识库 | 设置名称、类型、标签、简介、可见性（公开/受限）、Embedding 模型等 |
| 编辑知识库 | 修改知识库元信息 |
| 删除知识库 | 软删除（可恢复）或物理删除 |
| 知识库列表 | 分页展示，支持按名称/类型/标签筛选 |
| 知识库详情 | 查看文档数量、分段数量、最近构建时间、构建状态、失败原因与访问范围 |
| 权限配置 | 配置哪些用户/角色可以访问/管理该知识库 |
| 重新向量化 | 按修改后的分段规则对知识库内所有文档重新向量化 |
| 向量化进度 | 实时展示向量化任务进度 |

#### 5.4.2 知识库元信息

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 唯一标识 |
| name | string | 知识库名称 |
| type | enum | 类型（如：技术文档、产品手册、FAQ、通用知识等） |
| tags | string[] | 标签列表 |
| description | text | 简介/描述 |
| visibility | enum | 可见性（public 公开 / restricted 受限） |
| embedding_model | string | 使用的 Embedding 模型名称 |
| chunk_size | int | 默认分段大小（字符数） |
| chunk_overlap | int | 默认分段重叠（字符数） |
| status | enum | 状态（active / vectorizing / archived / deleted） |
| current_index_version | string | 当前索引版本号 |
| creator_id | UUID | 创建者 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 最后更新时间 |

#### 5.4.3 重新向量化流程

```
管理员修改分段规则 -> 点击"重新向量化"
  -> 创建变更快照
  -> 后端创建异步任务 -> 更新知识库状态为 "vectorizing"
  -> 逐文档按新规则分段 -> 重新生成 Embedding -> 写入新索引版本
  -> 校验成功后以原子方式切换当前索引版本
  -> 完成后更新状态为 "active"，记录操作日志
  -> 失败时保留上一个可用版本
```

**关键约束：** 构建应以后台任务执行，不能中断在线问答；重建成功后以原子方式切换当前索引版本；失败时保留上一个可用版本。在线旧索引在新版本发布前仍可用。

| 功能 | 说明 |

| 重新分段 | 修改规则后对单个文档重新分段和向量化 |

#### 5.5.2 上传与处理流水线

```

  -> 创建变更快照
  -> 后台向量化/建立全文索引
  -> 校验成功后发布新索引版本
  -> 写审计日志、指标与 Langfuse 追踪
```

每一步都保存处理状态和失败原因。

#### 5.5.3 文档状态流转

```
已上传 (uploaded) -> 解析中 (parsing) -> 预处理 (processing) -> 待确认分段 (pending_segment) -> 向量化中 (vectorizing) -> 已发布 (ready) -> 已归档 (archived)
                                                    |
                                                    v
                                         失败 (error，可重试)
```

| file_size | int | 文件大小（字节） |
| file_path | string | MinIO 对象存储路径 |

| creator_id | UUID | 上传者用户 ID |

### 5.6 智能问答与会话记忆

#### 5.6.1 功能点

| 功能 | 说明 |
|------|------|
| 多轮对话 | 基于会话上下文的多轮问答，保留对话历史 |
| 多用户并发 | 支持多用户同时进行问答，会话完全隔离 |
| 用户记忆 | 每个用户维护独立的对话记忆（短期上下文 + 长期摘要） |
| 流式输出 | SSE（Server-Sent Events）实时推送回复 |
| 来源引用 | 回答中标注信息来源文档和段落编号 |
| 相关性评分 | 展示检索结果的相关性得分 |
| 会话管理 | 会话新建、重命名、查看和删除本人历史会话 |

#### 5.6.2 检索策略

| 策略 | 说明 | 适用场景 |
|------|------|--------|
| 向量检索 | 基于 Embedding 的语义相似度检索（Chroma 原生） | 通用语义搜索、模糊意图 |
| 全文检索 | 基于关键词匹配的检索，利用 PostgreSQL `tsvector` 或应用层 BM25 实现 | 精确关键词匹配、法规条文检索 |
| 混合检索 | 向量 + 全文双路检索，应用层加权融合排序（RRF） | 综合场景，兼顾语义与关键词（推荐默认） |

说明：Chroma 本身专注于向量相似度检索，全文检索和混合排序在应用层实现。全文检索可选用 PostgreSQL 内置的 `tsvector` / `tsquery`（免额外组件），也可引入轻量级 Python 库实现 BM25 检索。

#### 5.6.3 问答流程

回答由"查询改写/上下文记忆 — 权限过滤 — 检索 — 重排 — 生成 — 引用返回"组成：

```
用户问题 + 会话标识
  -> 身份认证与知识库权限过滤
  -> 加载本会话上下文和摘要记忆
  -> 查询改写
  -> 多路检索（向量 + 全文）
  -> 结果融合排序（RRF / 加权）
  -> 相关性过滤（阈值截断）
  -> 上下文组装（拼接检索结果 + 对话历史）
  -> 基于证据生成流式回答
  -> 返回引用、追踪标识与性能数据
  -> 保存本用户会话历史和可观测记录
```

返回内容包含：答案、来源片段、检索方式、耗时和请求追踪标识。无法命中时明确说明未在授权知识库中找到依据，不得编造来源。

#### 5.6.4 会话管理

| 机制 | 说明 |
|------|------|
| 会话创建 | 访客首次提问时自动创建临时 session；注册用户登录后继承历史会话 |
| 上下文窗口 | 保留最近 N 轮对话（默认 10 轮） |
| 记忆摘要 | 超过上下文窗口的历史对话自动压缩为摘要，摘要附在新上下文中 |
| 会话过期 | 闲置超过 T 分钟自动过期（默认 30 分钟） |
| 会话切换 | 注册用户支持开启新会话或切换历史会话；访客仅当前会话，无历史切换 |
| 访客会话 | 使用匿名临时标识，关闭或过期后清理；不保存为可跨设备恢复的长期记忆 |

每次请求携带会话标识，服务端仅加载该用户可见的最近上下文和摘要记忆；不同用户、不同会话不得互相读取。检索范围根据当前用户授权的知识库计算；用户可在有权限时指定一个或多个知识库，默认使用全部可访问范围。

#### 5.6.5 并发控制

- 使用 Redis 管理会话状态，确保会话在请求间持久化
- 每个用户的会话独立存储，互不干扰
- LLM 调用采用 FastAPI 异步处理，避免阻塞其他用户请求
- Chroma 采用客户端-服务端模式部署，支持多连接并发读取

### 5.7 检索与命中率测试

命中率测试用于验证检索效果，不直接等同于模型回答质量。

#### 5.7.1 功能点

| 功能 | 说明 |
|------|------|
| 测试用例管理 | 创建/导入/编辑测试问题集（问题 + 期望文档/分段 + 备注） |
| 单题测试 | 选择知识库、文档、检索方式，输入测试问题，点击"运行测试" |
| 批量测试 | 选择测试问题集批量执行测试 |
| 测试结果展示 | 逐条展示问题、检索结果、是否命中、相关性得分、耗时、索引版本 |
| 命中率统计 | 总体命中率、Recall@K、MRR、平均耗时和失败样本 |
| 对比测试 | 同一问题集在不同策略下的命中率并排对比 |
| 结果导出 | 支持导出 CSV 格式的测试报告 |

#### 5.7.2 测试配置参数

| 参数 | 说明 | 可选值 |
|------|------|--------|
| 目标知识库 | 在哪个知识库中测试 | 支持多选，只能选择当前有权限的范围 |
| 目标文档 | 限定在哪些文档中测试 | 支持多选，留空表示全部 |
| 检索方式 | 使用的检索策略 | `vector` / `fulltext` / `hybrid` |
| Top-K | 返回的检索结果数量 | 1-20，默认 5 |
| 相似度阈值 | 最低相关性分数 | 0.0-1.0，默认 0.5 |

#### 5.7.3 命中率计算

```
命中率 = 命中问题数 / 总问题数 * 100%

指标至少包括：
- Recall@K：期望分段进入 Top-K 的比例
- MRR：平均倒数排名
- 命中率：以期望分段进入 Top-K 为准
- 平均耗时
- 失败样本列表

命中判定标准（可配置）：
- 精确匹配：检索结果中的文档 ID / 分段 ID 出现在期望列表中
- 语义匹配：检索结果内容与期望答案的语义相似度超过阈值
```

#### 5.7.4 测试约束

- 单题结果展示 Top-K 命中片段、文档名、分段序号、得分、耗时、索引版本与是否命中标准答案片段。
- 测试记录不可被索引重建覆盖，必须关联检索配置、模型版本和索引版本，便于横向比较。
- 混合检索应展示向量与关键词权重配置。

#### 5.7.5 测试结果数据结构

| 字段 | 说明 |
|------|------|
| question | 测试问题 |
| expected_doc_ids | 期望命中的文档 ID 列表 |
| expected_chunk_ids | 期望命中的分段 ID 列表 |
| actual_results | 实际检索结果（文档 ID + 分段序号 + 分数 + 内容摘要） |
| is_hit | 是否命中 |
| hit_rank | 命中的排名（Top-K 中的位置，未命中则为 null） |
| score | 相关性得分 |
| strategy | 使用的检索策略 |
| index_version | 测试时的索引版本 |
| elapsed_ms | 检索耗时（毫秒） |

### 5.8 快照、回退与审计

#### 5.8.1 功能点

| 功能 | 说明 |
|------|------|
| 自动快照 | 文档上传/删除/规范化/分段规则变更/批量重分段/向量化发布/知识库权限变更前，自动创建可恢复快照 |

| 回退操作 | 将知识库恢复到指定快照状态 |
| 快照清理 | 删除不需要的旧快照，释放存储空间 |

#### 5.8.2 快照内容

快照包含知识库元数据、文档版本、分段版本、索引版本、分段规则和权限配置引用。向量数据本身不直接存储在快照中，回退时通过文档元数据重建向量索引。

#### 5.8.3 回退流程

```
管理员选择知识库或文档快照
  -> 查看版本差异和影响范围
  -> 二次确认
  -> 自动创建"回退前保护快照"（安全保护）
  -> 以快照内容创建新的恢复版本（不覆盖原历史）
  -> 重建受影响索引
  -> 原子切换当前版本并记录审计日志
```

**关键约束：** 恢复操作生成一条新的索引版本，不覆盖原历史。

#### 5.8.4 快照策略

| 策略项 | 默认值 | 说明 |
|--------|--------|------|
| 最大快照数量 | 50 | 每个知识库最多保留的快照数，超出自动清理最早快照 |
| 自动快照触发 | 文档变更时 | 上传/删除/重新分段/重新向量化/权限变更操作前触发 |
| 快照保留天数 | 90 天 | 超过天数的快照自动清理 |
| 回退保护快照 | 始终创建 | 回退前自动创建保护快照，确保可撤销回退操作 |

#### 5.8.5 审计日志

所有关键操作写入审计日志：操作者、时间、对象、前后版本、请求标识、结果与失败原因。审计日志存储于 PostgreSQL（audit_logs 表）。

## 7. 数据模型

### 7.1 核心实体关系

```
User (用户)
  |--N:M-- Role (角色)
  |           |--N:M-- Permission (权限)
  |--1:N-- Session (会话)
  |           |--1:N-- Message (消息)
  |           |--1:1-- MemorySummary (记忆摘要)
  |--1:N-- Document (上传的文档)
  |
KnowledgeBase (知识库)
  |--1:N-- Document (文档)
  |           |--1:N-- Chunk (分段)
  |--1:N-- IndexVersion (索引版本)
  |--1:N-- Snapshot (快照)
  |           |--1:N-- SnapshotDocument (快照文档关联)
  |--N:M-- User / Role (访问权限，通过 kb_permissions 关联)
  |
HitTestRun (命中率测试运行记录)
  |--N:1-- HitTestCase (测试用例集)
  |--1:N-- HitTestResult (测试结果)
  |
ModelConfig (模型配置)
AuditLog (审计日志)
```

### 7.3 模块边界与约束

| 模块 | 核心对象 | 对外职责 |
|------|---------|---------|
| 认证与 RBAC | User、Role、Permission、kb_permissions | 认证、授权、资源范围判定 |
| 会话问答 | Session、Message、MemorySummary、Citation | 多轮会话、检索编排、回答与引用 |
| 知识库 | KnowledgeBase、Tag、IndexVersion | 隔离边界、元数据、发布版本 |
| 文档处理 | Document、DocumentVersion、Chunk、ChunkRule、ProcessingTask | 上传、解析、规范化、分段、向量化任务 |
| 检索评测 | RetrievalConfig、TestCase、TestRun、TestResult | 单题/批量评测与指标计算 |
| 快照审计 | Snapshot、RestoreJob、AuditLog | 版本保护、回退、操作追溯 |
| 模型与观测 | ModelConfig、TraceReference、Metric | 配置引用、Langfuse、Prometheus/Grafana 指标 |

模块间约束：
- 认证与 RBAC 是所有资源接口的前置依赖。
- 文档处理只发布索引版本，不直接修改会话。
- 问答仅消费"已发布"的索引版本。
- 快照与审计横切写操作。
- 测试模块必须固定其执行时的检索配置和索引版本。

## 11. 验收标准

### 11.1 最小验收场景

1. **权限隔离**：未注册用户只能对公开知识库问答，无法查看受限内容、上传文档或调用管理接口。
2. **知识库隔离**：两个不同角色访问同一问题时，检索来源严格限制在各自被授权知识库内。
3. **会话隔离**：两名登录用户同时进行多轮问答，A 的历史和上下文不会出现在 B 的回答中。
4. **文档处理**：上传 PDF、DOCX、TXT、Markdown 后可以看到解析、预处理、分段和索引状态；失败可见明确原因。
5. **重分段**：修改分段规则后，能预览分段差异、触发重分段和重向量化，在线旧索引在新版本发布前仍可用。
6. **命中率测试**：管理员可按知识库、文档和三种检索方式运行测试，结果包含 Top-K 片段、分数、耗时和索引版本。
7. **快照回退**：恢复某个快照后，知识库内容与索引切换到恢复后的新版本，并保留恢复前版本和审计记录。
8. **Docker 部署**：Docker Compose 启动后，FastAPI、前端、Langfuse、Prometheus、Grafana 均可访问；关键请求可关联到日志/追踪与指标。

### 11.2 命中率指标验收

- 批量测试集运行后，指标至少包括 Recall@K、MRR、命中率、平均耗时和失败样本。
- 测试记录关联检索配置、模型版本和索引版本，不被索引重建覆盖。

## 12. 待确认项

以下决策会影响后续实现，应在开始分工前确认：

1. **容量指标**：单文件大小上限、单知识库文档数上限、并发用户数、会话历史保存期限。
2. **命中标准**：以期望分段进入 Top-K 为准，还是以人工评分/答案正确性共同判定。
3. **访客策略**：公开知识库和访客会话的保留策略，以及上传文档的安全扫描要求。
4. **删除策略**：文档删除是否采用软删除、快照保留期限、可恢复范围与存储成本上限。
5. **模型清单**：LLM、嵌入模型、重排模型的本地/远程部署方式与具体模型清单（当前以 OpenAI 兼容接口为默认，可替换为本地部署模型）。
6. **重排模型**：是否引入独立的 Rerank 模型，还是使用应用层加权融合。

合并过程中的关键决策：
1. **权限标识统一**：采用冒号分隔格式（`kb:read`），合并两份文档的权限清单，补充 `snapshot:restore`、`model:read/write`。

3. **技术选型确认**：V1.0 待确认的中间件选型，以 V1.2.1 的 Chroma/MinIO/Redis/PostgreSQL 为确认方案。

5. **命中率测试**：合并 V1.0 的 Recall@K/MRR 指标要求与 V1.2.1 的测试配置参数和结果数据结构。
6. **新增模块**：补充大模型管理模块（V1.0 有产品要求，V1.2.1 无对应 API），新增审计日志 API。
7. **数据模型增强**：新增 IndexVersion、MemorySummary 实体，反映 V1.0 的索引版本管理和记忆摘要概念。
